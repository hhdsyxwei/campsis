# Campsis量化交易系统（智能厨房版）架构总览

## 一、架构设计背景
### 1. 目标用户
个人量化交易者、小型交易团队（3-10人）

### 2. 核心痛点
- 大型机构量化系统复杂、部署成本高，不适配小体量用户；
- 零散工具（数据下载、回测、下单）割裂，缺乏全流程闭环；
- 非专业开发人员上手门槛高，风控意识薄弱。

### 3. 设计灵感
以「智能厨房」为核心隐喻，复刻「备料→做菜→出餐→复盘」的生活化闭环逻辑，打造轻量化、易上手、全流程可控的量化交易系统。

## 二、架构层级关系（自上而下/核心流程）
采用**分层解耦+闭环反馈**的结构，核心正向流程为「底层支撑 → 数据处理 → 核心能力 → 策略研发 → 交易执行 → 人机交互」，反向通过反馈链路优化策略/数据，整体逻辑如下：

### 核心层级链路（正向）
1. 厨房基座层（Kitchen Base Layer）

   └── 为所有层级提供基础支撑（配置、日志、异常处理）
   
   └── 输出 → 食材管理层
2. 食材管理层（Ingredient Layer）

   └── 处理量化数据全生命周期
   
   └── 输出 → 智能烹饪层
3. 智能烹饪层（Cooking Engine Layer）

   └── 提供回测/风控等核心引擎能力
   
   └── 输出 → 菜谱研发层
4. 菜谱研发层（Recipe R&D Layer）

   └── 研发/验证交易策略
   
   └── 输出 → 出锅执行层
5. 出锅执行层（Serving Layer）

   └── 将策略落地为模拟/实盘交易
   
   └── 输出 → 餐桌交互层
6. 餐桌交互层（Dining Table Layer）

   └── 展示交易结果+用户交互

### 反向反馈链路
- 餐桌交互层 → 菜谱研发层：反馈优化策略参数/逻辑
- 餐桌交互层 → 食材管理层：反馈优化数据质量
- 所有层级 → 厨房基座层：汇总日志/异常/配置变更


## 三、层级交互关系说明（文字拆解）
### 1. 正向核心链路（主流程）
- **厨房基座层 → 所有层级**：为全系统提供「水电燃气级」基础支撑，比如给食材层提供数据源配置、给执行层提供异常重试能力、给交互层提供日志查询接口，无业务逻辑，仅保障系统能运行。
- **食材管理层 → 智能烹饪层**：输出标准化的「干净食材（数据）」，比如清洗后的K线数据、财务数据，供烹饪层（引擎）做回测、行情分析。
- **智能烹饪层 → 菜谱研发层**：提供「烹饪工具（引擎能力）」，比如回测引擎帮菜谱层（策略）模拟“试做菜”，风控引擎帮菜谱层校验策略风险。
- **菜谱研发层 → 出锅执行层**：输出「定稿菜谱（成熟策略）」，比如经过回测优化的均线策略信号，供执行层落地为模拟/实盘交易。
- **出锅执行层 → 餐桌交互层**：推送「做好的菜品（交易结果）」，比如实盘成交记录、账户盈亏数据，供交互层展示和告警。

### 2. 反向反馈链路（优化流程）
- **餐桌交互层 → 菜谱研发层**：用户通过交互层复盘发现「菜品不好吃（策略收益差）」，反馈给菜谱层调整策略参数/逻辑（比如修改均线周期）。
- **餐桌交互层 → 食材管理层**：交互层发现「食材不新鲜（数据异常）」，反馈给食材层重新清洗/补充数据（比如重新拉取停牌期间的行情）。
- **所有层级 → 厨房基座层**：各层级运行中的日志、异常、配置变更，均汇总到基座层存储/处理（比如执行层下单失败的异常日志，基座层记录并触发重试）。

### 3. 跨层核心交互规则
1. **接口标准化**：层与层之间仅通过统一接口通信（比如食材层只输出DataFrame格式数据，烹饪层仅调用接口获取，不直接操作食材层数据）；
2. **风控全嵌入**：智能烹饪层的风控引擎贯穿所有执行环节——执行层下单前需过风控校验、订单运行中风控实时监控、成交后风控复盘亏损原因；
3. **事件解耦**：智能烹饪层的事件总线作为“消息中枢”，比如行情更新事件同步给菜谱层（触发策略信号）、执行层（触发下单），避免层级间直接依赖。

## 四、层级核心定位速览（文字版）

|----------------|-------------------------|--------------------------|------------------------------|-----------------------------------|
| 厨房基座层     | Kitchen Base Layer      | 系统底层基础支撑         | 水电燃气、橱柜、基础厨具     | 配置管理/日志/异常处理/数据备份   |
| 食材管理层     | Ingredient Layer        | 数据全生命周期管理       | 买菜、洗菜、存菜、取菜       | 数据源接入/数据清洗/存储/查询     |
| 智能烹饪层     | Cooking Engine Layer    | 核心引擎能力（回测/风控）| 智能炒锅、火候控制、烟雾报警 | 回测/行情/风控/绩效分析/事件总线  |

## 五、架构核心设计原则
1. **轻量化优先**：单设备部署，Python轻量化技术栈，无需复杂运维（如微服务、分布式架构）；
2. **解耦不拆分**：层间接口标准化，但模块内部轻量化整合，降低维护成本；
3. **风控前置**：风控引擎嵌入全流程，避免策略bug或接口异常导致重大亏损；
4. **易用性至上**：策略模板化、操作可视化，降低非专业开发人员使用门槛；
5. **开源复用**：核心引擎优先复用成熟开源库（Backtrader、VNPY、Tushare），不重复造轮子。

## 六、总结
1. 整体架构以「智能厨房闭环」为核心，6层职责边界清晰，正向完成「数据→策略→交易」的核心流程，反向通过反馈优化策略/数据；
2. 厨房基座层是全系统的基础支撑，智能烹饪层是核心能力枢纽，餐桌交互层是用户反馈的核心入口；
3. 层间通过标准化接口+事件总线解耦，既保证轻量化，又支持模块单独升级/替换（如更换数据源、切换券商接口）；
4. 适配个人/小型团队核心诉求：低成本部署、易上手操作、全流程风控可控。
